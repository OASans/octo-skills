Commit staged and unstaged changes with a conventional commit message. No pull, no build, no tests — just commit.

## Commit Message Rules

- **CRITICAL: No AI fingerprints.** Never include AI-generated signatures, attributions, or markers. This includes:
  - "Generated by", "Created by", or any tool attribution
  - "Co-Authored-By" any AI (Claude, GPT, Copilot, etc.)
  - Emojis of any kind (robot, sparkles, or otherwise)
  - Any other text indicating AI involvement
- Messages must read as if written by a human developer.
- Use conventional commits: `type(scope): description` (e.g. `feat(ipc): add click command`, `fix(layout): correct grid overflow`).
- Types: feat, fix, refactor, test, docs, chore, perf, style, ci, build.
- Scope: module or area affected. Omit if change is broad.
- Description: imperative mood, lowercase, no period.
- Body: bullet points summarizing what changed at a high level, with enough detail to understand the scope.
- **Batch trailer.** If `$ARGUMENTS` contains a tracker path, add a `Batch: <tracker-filename>` trailer at the end of the commit message (after the body, separated by a blank line). This links the commit to its parent batch run.

## Steps

1. Run `git diff` and `git diff --cached` to collect all changes. If both empty, say "Nothing to commit." Stop.
2. Stage all unstaged changes with `git add` (specific files, not `-A`).
3. Analyze the diff. Write a conventional commit message following the rules above.
4. Create the commit. Use a HEREDOC for the message. If `$ARGUMENTS` was provided, include the `Batch:` trailer.
